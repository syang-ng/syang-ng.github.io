SOSP 2019 TSVD

Written by syang with ♥ on April 10, 2020 in paper

SOSP - 2019 - Efficient Scalable Thread-Safety-Violation Detection - Finding thousands of concurrency bugs during testing

论文链接：https://www.microsoft.com/en-us/research/uploads/prod/2019/09/sosp19-final193.pdf

项目地址：https://github.com/microsoft/TSVD

会议 PPT：https://sosp19.rcs.uwaterloo.ca/slides/li.pptx

本篇文章是 SOSP 2019 两篇 best paper 之一，出自芝加哥大学的 Shan Lu 教授所带领的研究组，对于 Shan Lu 教授感兴趣的可以查看她的个人主页：http://cs.uchicago.edu/~shanlu/

主要内容

一般来说，并发漏洞是非常难以查找、重现和调试的，而且在内部测试中其实也很难找到并发漏洞，但并发漏洞一旦引爆，极有可能造成生产环境的大规模停机等糟糕后果。现有的并发漏洞检测工具不能很好地集成到现有测试环境中的原因在于没有很好地解决以下几个问题：如何处理数千个使用不同同步机制团队开发的代码？如何降低工具的误报？如何尽可能的少占用测试资源？

为了解决上诉问题带来的挑战，本篇工作提出了 TSVD，一款全新的并发漏洞检测工具，不同于先前工作随机注入延迟时间或者是采用代价昂贵的同步检测机制，TSVD 提出了一种轻量级的 near-miss tracking 机制来识别可能存在并发漏洞的线程不安全函数，然后结合一种作者称为 happens-before(HB) inferencing 的机制，TSVD 能在测试过程中有效地发现存在并发漏洞的调用或者排除不可能的调用。

作者在 .NET 平台上实现了 TSVD 的原型，包括两部分：TSVD runtime library，实现了 TSVD 核心算法；TSVD instrumenter，能通过静态插桩的方式在 .NET 应用中集成 TSVD runtime library。最终 TSVD 找到了 1134 个并发漏洞，其中 80 个漏洞经过了开发者的确认，包括 77 个开发者之前未发现的并发漏洞。

背景知识

TSV(Thread-Safety Violation)

一般来说，一种程序语言的类库会定义线程安全相关的规则来决定其是否能被并发调用，而如果这种规则被违反，则我们称之为 Thread-Safety Violation。

以类 Dictionary 为例，读操作 ContainsKey 可以被多线程同时调用，但写操作 Add 则只能在一个互斥的上下文内完成，看下面这段代码，线程 1 访问的 key1 和线程 2 访问的 key2 可能是同一个变量，因此这里存在着 TSV。

// Dictionary dict
// Thread 1:
dict.Add(key1, value)
// Thread 2:
dict.ContainsKey(key2)

happens-before

这里介绍一下 Java 中的 happens-before：

  1. 如果一个操作 happens-before 另一个操作，那么第一个操作的执行结果将对第二个操作可见，而且第一个操作的执行顺序排在第二个操作之前。
  2. 两个操作之间存在 happens-before 关系，并不意味着 Java 平台的具体实现必须要按照 happens-before 关系指定的顺序来执行。

happens-before 原则可以用来判断数据是否存在竞争、线程是否安全。

Inject Delay

有没有一种能降低 false positive 的方法？当然有，只要我们在 bug 触发的时候定位到这个 bug，那当然不存在任何的 false positive。那么如何触发 bug？一个很直接的思路就是向一个线程注入一定的延迟，增加两个线程同时访问一个变量，触发并发漏洞的可能性，示意图如下：

已有工作的缺点

已有工作存在着很大的 overhead，其原因主要有以下两点：

  1. 比如 DataCollider 会在所有可能的地方插入 delay，但需要多次运行以触发 bug，同时需要插入 delay 的位置多了本身就会带来巨大的 overhead
  2. 第二种思路，如 RaceFuzzer 则是使用先动静态结合的分析工具，找到合适的插入 delay 的位置再精准插入。虽然该方法会减少总共需要插入 delay 的数目，但在分析阶段的 overhead 同样也很大。

如下图所示，不同的工具在不同的侧重点上有着很大的 overhead，而 TSVD 的设计目的，就是尽可能地降低在识别和运行过程中的 overhead。

设计实现

说了这么多，下面来仔细介绍以下 TSVD 的实现。首先，TSVD 会用一个轻量级的分析工具定位出所有可能造成 TSV 的调用点，称为 trap set，随后会在这些调用点进行插桩，所以这些函数在执行时就会调用 OnCall 函数。

该机制的具体逻辑是：当第一个线程调用该函数时，TSVD 会在第 3 行告诉该线程是否需要 delay，如果需要，则在第 4 行将该线程注册到一个全局的 table，然后 delay。当第二个线程调用该函数时，会在第 2 行检查此前是否已经有 delay 的线程与它相关。检查的方式是判断两个线程的 thread_id 是否不一样，而 obj_id 是否一样，两个 op_id 是否至少有一个是 write。如果以上要求都满足的话，则找到一个并发漏洞。

由于该机制在设计上很明显不会有任何的 false positive，唯一的挑战在于，如何有效地判断何时需要 delay，因为无条件的 delay 很明显会有很大的 overhead。这里作者提出了两个巧妙的思路来判断何时可能需要 delay，何时可能不存在 TSV。

Identifying near misses

该思路其实就是通过 close-by physical 来找到 concurrent logical，我们知道并发漏洞出现在多线程对一个变量的访问上，而逻辑上的并发在时间上的表现就是重合或者是时间相近，因此如果两个线程对同一个变量的访问在时间上相近，而且其中有一个是写操作的话，就可以把它标记为需要 delay 的调用点。

Inferring likely HB relationship

前一阶段找到确认的两个调用点虽然在时间上有联系性，但这样不能代表绝对会出现 TSV，比如 happens-before 关系就是一个反面例子，所以此时需要去掉这些错误标记的调用点。这里的思路也非常直接巧妙，我们知道虽然不同的同步机制非常多，实现也各有不同，但如果同步机制确实存在的话，那么对 thread 1 的 delay 同样也会造成对 thread 2 的 delay，因此我们可以利用这种逻辑上的联系帮我们确定是否存在可能的 happens-before 关系。

比如我们在 thread 1 插入了一定的延迟，如果两个线程的访问存在着 HB 关系的话，可以看到 thread 2 也同样地发生了一定比例的延迟。

实验评估

Overall

实验使用的 benchmark 如下：

  • Large benchmark：微软开发的 1657 个项目中使用的 43000 个模块，包含了约 122000个多线程单元测试和 89000 个二进制文件
  • Small benchmark：从 Large benchmark 中随机选取的 1000 个模块

其中 Large benchmark 用一台配置有Intel(R) Xeon(R) E5-2650 CPU，128G memory 以及 6T SSD 的服务器进行测试，而 Small benchmark 用一台搭载 Intel(R) Xeon(R) E5-1620 CPU，16G Memory 和 1T SSD 的服务器进行测试。

最终找到了 1134 个 TSV，其中 96% 是之前未知的 bug，47% 会导致严重的问题。

在开源数据集上的评估

为了证明 TSVD 也可以应用到除了 Microsoft 外的其它开源 C# 项目上，作者从 Github 上随机选取了 9 个开源项目进行测试，结果同样证明了 TSVD 的高效准确。

总结

作为 SOSP 19 的 best paper，TSVD 确实提个两个 idea 都非常有意思而且高效可行，最后也取得了很好的效果。不过这里注意到其实 TSVD 还是牺牲了可靠性来保证了完备性的，即我不保证找到全部的 bug，但我确认我找到的 bug 都是对的，而且我尽可能多的找到了 bug。这里有一个值得思考的点是，其实工业界有可能更看重可靠性，虽然有些工具在学术研究过程中，跑几小时甚至一两天都是合理的，但在工业界的代码量级和需求下，更快开销更低可能更贴合实际。
